git rebase

Replay commits onto a new base — for linear history, cleaned-up branches, and rewriting local work before it ships.

Beginner — the mental model

What rebase does

Takes your branch's commits, lifts them off, and replays them one by one on top of another commit. Result: same changes, new history.

git rebase main

Replay current branch onto the tip of main. Your commits get new hashes — they are copies, not moves.

Rebase vs merge

Merge ties two histories together with a merge commit. Rebase rewrites yours to look like it started from the new base — linear, no knot.

Commits are snapshots

Each replayed commit is re-applied as a diff. If the new base conflicts with that diff, git pauses and asks you to resolve.

The golden rule

Rebase only commits that exist on your machine only. Never rebase history others have already pulled.

Typical flow

git switch feature
git fetch origin
git rebase origin/main
# resolve conflicts if any
git push --force-with-lease

Practical — daily commands

git rebase -i HEAD~3

Interactive: edit the last 3 commits. Opens a todo list — reorder lines, change pick to an action.

Interactive actions

pick keep · reword edit message · squash merge into previous (keep msg) · fixup merge, drop msg · drop delete · edit pause to amend.

Continue / bail out

git rebase --continue
git rebase --skip
git rebase --abort

--abort restores the branch exactly as it was before the rebase started.

Squash a whole branch

git rebase -i main

Mark every commit but the first as fixup → one clean commit for review.

Pull without merge knots

git pull --rebase
git config --global pull.rebase true

Autosquash workflow

git commit --fixup abc123
git rebase -i --autosquash main

Fixup commits auto-slot under their targets in the todo list.

Push rebased work

git push --force-with-lease

Refuses to push if someone else pushed meanwhile. Never plain --force.

Pitfalls — footguns & fixes

Rebasing shared commits

Rewrites history teammates already have — their next pull creates duplicate commits and conflicts. Only rebase unpushed or solo-branch work.

"I lost my commits"

You didn't. Reflog keeps everything ~90 days:

git reflog
git reset --hard HEAD@{2}

Same conflict, every commit

Rebase replays commits one at a time, so one conflicting change can conflict repeatedly. Enable rerere to auto-reuse resolutions:

git config --global rerere.enabled true

Force-push over teammates

Plain --force silently deletes others' pushes. --force-with-lease aborts instead — make it a habit.

Empty commits vanish

If a commit's changes already exist upstream, rebase drops it silently. Use --keep-empty / check with git log after.

Mid-rebase confusion

git status tells you you're mid-rebase and which commit is being applied. Lost? git rebase --abort and start over — it's free.

Expert — power moves

--onto: transplant a branch

git rebase --onto main topicA topicB

Move commits reachable from topicB but not topicA onto main. Precise branch surgery.

--update-refs

Rebasing a stack of branches? Git ≥2.38 moves every branch pointer in the stack in one rebase. Essential for stacked-PR workflows.

--exec: test every commit

git rebase -i --exec "make test" main

Runs the command after each replayed commit; stops on first failure. Keeps every commit green for bisect.

--rebase-merges

Preserves merge-commit topology while rebasing instead of flattening it. For rebasing long-lived integration branches.

Split a commit

Mark it edit, then:

git reset HEAD^
git add -p; git commit
git rebase --continue

--committer-date-is-author-date

Keeps original timestamps when replaying — history reads naturally after big cleanups.

Autostash

git config --global rebase.autoStash true

Dirty tree? Git stashes, rebases, pops — no manual stash dance.

Alternatives — when not to rebase

git merge

Safe for shared branches; keeps true history. Pick merge when the branch is public or the history knot doesn't bother you.

Squash-merge (PR button)

GitHub/GitLab squash on merge gives you the clean linear main without anyone rebasing locally. Most teams' default.

git cherry-pick

Replay specific commits elsewhere instead of a whole branch. Rebase is cherry-pick in bulk.

git commit --amend

Only fixing the latest commit? Amend is a one-commit rebase with none of the ceremony.

jj / Sapling / Graphite

Newer VCS layers make history rewriting first-class and undoable — worth a look if you live in stacked diffs.

Rule of thumb

Rebase to clean up before sharing; merge (or squash-merge) to combine after sharing.
