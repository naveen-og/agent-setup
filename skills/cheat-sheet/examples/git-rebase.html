git rebase: Rewriting History Safely

Clean, linear commits before your work ships — without losing anything on the way.

The golden rule, first. Rebase only commits that exist on your machine alone. Never rebase history other people have already pulled — everything else in this sheet is safe as long as you hold this line.

The one idea

Rebase is replaying commits onto a new base. Git lifts your branch's commits off, then re-applies them one by one on top of another commit. Same changes, new history — every replayed commit gets a new hash, because it's a copy, not a move.

main:      A──B──C
feature:         └──X──Y        git rebase main
result:    A──B──C──X'──Y'      (X, Y replayed as copies)

Compare with merge: a merge ties the two histories together with a merge commit and keeps the knot. Rebase rewrites yours to look like it started from the new base — linear, no knot. If a replayed commit conflicts with the new base, git pauses and asks you to resolve before continuing.

The commands you actually need

git rebase main               # replay current branch onto main's tip
git rebase -i HEAD~3          # interactive: edit the last 3 commits
git rebase --continue         # after resolving a conflict
git rebase --abort            # bail out — restores branch exactly as before
git pull --rebase             # update without merge knots
git push --force-with-lease   # push rebased work (never plain --force)

Interactive rebase opens a todo list. Reorder the lines, or change pick to an action: reword edits the message, squash merges into the previous commit keeping both messages, fixup merges and drops the message, edit pauses so you can amend, drop deletes.

The tidiest workflow is autosquash — commit fixes as you go, fold them in at the end:

git commit --fixup abc123          # mark this commit as a fix for abc123
git rebase -i --autosquash main    # fixups slot under their targets

Where people get burned

Rebasing shared commits. Rewrites history teammates already have — their next pull produces duplicate commits and conflicts. This is the golden rule again; it's the only rebase mistake that hurts someone other than you.

"I lost my commits." You didn't. The reflog keeps everything for about 90 days:

git reflog                    # find the commit you were on
git reset --hard HEAD@{2}     # go back there

The same conflict, every commit. Rebase replays commits one at a time, so one conflicting change can conflict repeatedly. Turn on rerere once and git reuses your resolutions: git config --global rerere.enabled true

Force-pushing over a teammate. Plain --force silently deletes whatever they pushed while you rebased. --force-with-lease refuses instead — make it a habit, there is no reason to ever type the plain form.

Power moves

git rebase --onto main topicA topicB   # transplant topicB's own commits onto main
git rebase -i --exec "make test" main  # run tests after every replayed commit
git rebase --update-refs main          # move a whole stack of branches (git ≥ 2.38)
git config --global rebase.autoStash true   # dirty tree? stash, rebase, pop — automatic

To split a commit: mark it edit in the todo, then git reset HEAD^, commit the pieces separately with git add -p, and git rebase --continue. And if you rebase long-lived branches that contain merges, --rebase-merges preserves the merge topology instead of flattening it.

When not to rebase

Rebase to clean up before sharing; merge to combine after sharing. For a public branch, git merge is the honest tool — it keeps true history and never breaks anyone. Most teams get the clean linear main anyway by using the squash-merge button on the PR, which means nobody has to rebase locally at all.

Smaller jobs have smaller tools: git commit --amend is a one-commit rebase with none of the ceremony, and git cherry-pick replays specific commits somewhere else — rebase is just cherry-pick in bulk.
